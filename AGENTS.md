# 開発ルール

## コメント方針
- コメントは日本語で書く。
- なるべくコメントを書く。
- 特に「意図」「前提」「制約」「ハマりどころ」がある箇所は必ずコメントを残す。
- 自明な処理の説明より、判断理由が伝わるコメントを優先する。
- テストメソッド・テスト関数名は日本語で書く。

## 推奨ルール
- 1つのPR/コミットは1テーマに絞る（レビューしやすくするため）。
- コミットメッセージは日本語で統一する。
- 命名は略語を避け、役割が分かる単語を使う。

## コミュニケーション方針
- 会話・説明では、なるべく略語を使わない。
- 略語を使う場合は、最初に正式名称を書く（例: Frontend、Backend）。
- なるべく平易な言葉を使い、専門用語は必要最小限にする。
- 相手がすぐ判断できるように、曖昧な表現を避けて具体的に説明する。
- 常に、分かりやすいコミュニケーションスタイルを優先する。

## TDD方針
- 開発はTDDで進める（Red → Green → Refactor）。
- ドメインロジックは厳密TDDで実装する。
- UI見た目はTDDより「受け入れテスト + E2E」で担保する。
- 1コミット1意図で分け、Red→Green→Refactorの流れが追える粒度を維持する。

## TDD実行ルール
- 実装コードを先に書かない。必ず「失敗するテスト（Red）」を先に作る。
- すべての機能変更は、対応するテストの追加/更新をセットで行う。
- 作業完了報告では、Red / Green / Refactor の実施内容を明記する。
- 例外は「設定ファイル更新・コメント修正など、挙動に影響しない変更」のみとする。
